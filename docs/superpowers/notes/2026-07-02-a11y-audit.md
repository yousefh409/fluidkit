## 2. `Thinking` `role="status"` semantics

- `Thinking` sets `role="status"` and `aria-label={label}` (default `"Thinking"`,
  overridable via the `label` prop) on the `Droplets` root it renders.
- `aria-live` behavior: `role="status"` implies `aria-live="polite"` per the ARIA
  spec, so no explicit `aria-live` attribute is needed — this is the correct,
  minimal implementation.
- The label is present by default and fully overridable (`<Thinking label="Working" />`),
  tested in `tests/components/Thinking.test.tsx`.
- Cross-checked against the `Droplets` `aria-hidden` fix above: `Thinking` passes
  `role="status"` through `Droplets`' `...rest`, which now suppresses `Droplets`'
  default `aria-hidden` — verified by the existing `Thinking` test staying green.

**Verdict: already compliant**, no change needed beyond the `Droplets` interaction above.

## 5. No keyboard traps anywhere

- Pointer capture: `Droplets.interactive` calls `setPointerCapture` on the dragged
  drop's pointer id (captures the *pointer*, not keyboard focus) and never sets
  `tabindex`. `LiquidDrag` delegates dragging entirely to Motion's own `drag` gesture,
  which is pointer/touch-driven and likewise never touches `tabindex` or focus.
  Neither can trap keyboard navigation because neither is in the keyboard focus
  order to begin with (no `tabindex`, no native focusable element).
- Global scan (see item 3's "Global scan" row): no component in `src/` sets
  `tabindex > 0`, and the only `tabindex` values anywhere are `useTabList`'s roving
  `0`/`-1` pair, which is the correct WAI-ARIA roving-tabindex pattern, not a trap.
- `JellyButton` is a real `<button>`; disabling it natively removes it from the tab
  order via the browser's own `disabled` semantics, not a manual `tabindex` hack.

**Verdict: already compliant**, confirmed by direct source grep rather than inference.
